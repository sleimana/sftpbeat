# Introduction:

The Collector, a software service deployed on Logstash collectors, is purpose-built for secure log retrieval from a variety of remote data sources, leveraging the SFTP protocol. These remote nodes are equipped with configured SFTP servers, facilitating connections to segregated directories that host the log files.

In addition to ensuring uninterrupted availability through deployment on a redundant pair of Logstash machines, establishing robust high availability and failover mechanisms, the solution guarantees the confidentiality, integrity, and availability of the log files collected. This comprehensive approach maintains an exclusive single-instance operation at any given time, providing peace of mind for service security and continuity.
Furthermore, the system is equipped to send alerts to the SIEM when necessary, enhancing its proactive monitoring and response capabilities to prevent any possible data-loss on time.

# How does it work?

During the service startup, the collector main module carries out a series of steps. First, it reads a catalog file called "collector.list" which contains details about various data sources. Next, it connects to each data source in the order they are listed. The collector then searches for files within each data source based on predefined criteria. Once it finds files that meet the criteria, it downloads them. After downloading, the collector performs a set of actions on each file, including extracting its contents, checking its integrity, and, if necessary, sending copies to other connected systems.

This entire process repeats in cycles at regular intervals, determined by the connection_interval parameter, for all data sources listed in the catalog file. Files meeting the criteria are identified as those that haven't been collected before and were generated within the past "n" days, as specified by the date_range parameter. Additionally, the remote files must adhere to a specific naming convention outlined later in this document.

At the end of each cycle, the collector conducts several checks to ensure the integrity and completeness of the collected data. If any anomalies or issues are detected during these checks, the collector will generate appropriate alarms and send them to the Security Information and Event Management (SIEM) system if necessary. This proactive approach helps in early detection and response to potential data-loss problems, enhancing the overall reliability and security of the system.
The file operations and checks performed by the collectors encompass the following:

## File Operations:

Within the operational context, various file operations are executed upon successful log file retrieval, including the following main actions:

* -	File Signing: The system employs PKI cryptographic signatures, which are automatically applied to all collected files, except for those originating from hosts explicitly listed in the configuration file under the signing section. The signing and verification keys can also be located at the path indicated in the configuration file for easy access and management.

* -	File Extraction: Procedures for file extraction along with stringent size control mechanisms. In certain data sources, files are stored in a compressed format. These compressed files are extracted to perform integrity checks and facilitate indexing procedures.


* -	File Encryption: Implementation of file encryption protocols as needed. Files originating from SAG systems are encrypted using a secret key located in the application directory. It's crucial to note that the loss of this key would render it impossible to decrypt these files.

* -	Elasticsearch/Kibana Data Dispatch: Transmission of log copies to Elasticsearch for data indexing and visualization. Specifically, only copies of the raw file can be forwarded to Elasticsearch for this purpose. This functionality can be deactivated or configured according to preferences through the configuration file.

* -	Disaster Recovery (DR) in Catania: When enabled, copies of the files are securely sent to the Disaster Recovery system situated in Catania. This process ensures data availability, and resilience.


* -	WORM (Write Once, Read Many) Storage: Establishment of copies within Write Once, Read Many storage systems. Parameters can be found in configuration file under the NFS section. 

## Health/Integrity checks and SIEM alerting:

  The collector has the capability to transmit alerts to the SIEM as syslog messages. These predefined alerts primarily serve to avert data loss and ensure uninterrupted service operation. Additionally, a record of the dispatched alerts is stored in the file system, with the designated path specified by the parameters log_dir in the configuration file under the Alerting section.

# Service Configuration:

Before initiating the service, it is imperative to configure pertinent settings. Specifically, two essential files warrant configuration: the "collector.list" catalog file, delineating data sources, and the "collector.conf" configuration file, encompassing the service's operational parameters. These configurations underpin the effective functioning of the Collector service.

## Catalog file - collector.list:
The data source catalog file assumes a critical role in the collector data retrieval system. In CSV format, it lists all integrated data sources. This file serves as a repository of metadata crucial for the acquisition process, including connection details, remote directory and file search criteria. When adding a new entry to this file, the following aspects should be considered.
The columns are explained as following: 
* A - Host: Refers to the hostname of the remote data-source.
* B - IP: Denotes the IP address of the remote data-source, used for mainly connections, differential analysis and alerting.
* C - Path: Specifies the remote path where the pertinent files reside.
* D - File: Specifies the prefix assigned to files of interest with respect to the file naming convention. 
* E - Extension: Specifies the file extension for the targeted files.
* F - System: Pertains to the nomenclature of the folder established within the system's designated "hot" directory, delineated by the variable %hot_dir%.

## Configuration File - Collector.conf:

There are two files present: one named "collector.conf.reference" which serves as a reference file and contains explanatory comments for the parameters, and another named "collector.conf" which serves as the effective configuration file. The latter file, "collector.conf" includes the parameters without comments, enhancing visibility and clarity for configuration purposes.
The configuration file comprises distinct sections denoted by square brackets [ ] . These sections enable the configuration of specific aspects of the collector as follows:

* 1. [Collector]:
*    This section encompasses parameters pertinent to the collection service. It includes settings for SFTP communication, intervals for establishing connections with remote machines, failover timing between the two collector nodes, data source configuration, remote file management, and legacy system login settings.
* 2. [DR]:
*    The DR section contains parameters essential for authentication and the transmission of log copies to the disaster recovery machine, ensuring data resilience and continuity in adverse scenarios.
* 3. [Elasticsearch]:
*    This section contains parameters required for sending log copies to the Elasticsearch engine, facilitating indexing and robust log management and searching using Kibana (when applicable).
* 4. [NFS]:
*    The NFS section encompasses parameters necessary for sending logs to the Write Once, Read Many (WORM) storage, enhancing data retention and compliance.
* 5. [Signer]:
*    Within the Signer section, parameters are configured to enable log signing and verification, ensuring data integrity and authenticity.
* 6. [Alerting]:
*    This section contains parameters for managing alerts generated by the service. It facilitates alert management and transmission to the SIEM system for proactive service monitoring.

# Service Troubleshooting:

When troubleshooting a potential service malfunction, the initial step is to verify the service's operational status using the following command:
systemctl status collector
If the service is not currently running, it can be initiated with the following command:
systemctl start collector
It's imperative that the collector service is active and operational on both Logstash instances. The service itself ensures the correct application of the fail-over mechanism. To configure the failover mechanism, adjust the failover_interval parameter in the configuration file. Importantly, ensure that this parameter is set to a value greater than the connect_interval.

Hereâ€™s few possible problems to encounter:
## Remote file is not collected:
* -	Check connection with remote host and the credentials/keys parameters set in the configuration file. 
* -	Check the remote file exists and has the following name convention: file-prefix_host-name_date*.file-extention, however the search criteria is file-prefix*anything*Date.file-extention
> * o	Accepted date formats: %Y-%m_%d, %Y-%m-%d, %d%b%Y, %Y_%m_%d, %Y%m%d
> * o	Accepted file extensions: .log, .txt, .gz, .tar.GZ, .tar.Z
* -	Check the hot directory on the Logstash machine: /%hot_dir%/%year%/%system%/%hostname%/%month%/%day%
Important: The date within the path corresponds to the date of retrieval rather than the file's original creation date. This entails the potential scenario wherein folders on the remote data source might contain files generated at earlier dates, but these files remain uncollected until the retrieval date.

## File is not signed:
-	Check the configuration file, signer sections. As per requirements some files from specific sources should not be signed.

## File is not encrypted:
-	As per requirements some files from specific sources should not be encrypted, unless the file belongs to SAG systems ('20.0.0.1','20.0.0.2', '20.0.0.3').
Troubleshooting possible problems through debug:

To facilitate process tracing, it is essential to configure the logging level to debug in the configuration file, setting it to 10.
Subsequently, the log file output can be accessed in the directory: /data/collector/logs/ where can be found log files generated by both Logstash nodes.

It's important to be aware of potential error messages that could hinder the service's operation. The Collector may encounter operational challenges, including configuration errors, system resource limitations, or connectivity interruptions. When such issues arise, it is imperative to pay close attention to any error messages that may surface. To gain a deeper understanding of the situation and identify the root cause, it is essential to examine the log file. Within the log file, specific information (represented as '%s') is provided to assist in the troubleshooting process.

## Possible warning messages:
The log file may contain the following messages as warnings. While these warnings do not impact the service's immediate functionality, they provide valuable insights into potential future errors.
Setting the log level to 'debug' offers a comprehensive view of system activities, enabling the identification of the root cause of any issues that need resolution. This enhanced visibility is achieved through the inclusion of additional informative messages in the logs.

Moreover, the system generates log files that adhere to the retention policy its period specified in the configuration file. It's important to note that log rotation comes into effect when the log files reach their allocated maximum storage space. This ensures efficient log management and prevents unnecessary overuse of storage resources.
